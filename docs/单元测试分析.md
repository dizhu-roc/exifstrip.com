# 单元测试分析

## 一、当前状态

- **无现有测试**：项目内暂无 `*.test.ts` / `*.spec.ts`，也未配置 Jest、Vitest、Playwright 等。
- **技术栈**：Next.js 16、React 19、TypeScript；核心逻辑在 `src/lib/`、自定义 hook `useUploadFiles`。

---

## 二、测试步骤与顺序（三阶段）

建议按以下顺序落地，先验证「界面与流程」再深入「代码逻辑」：

| 阶段 | 名称 | 目标 | 典型手段 |
|------|------|------|----------|
| **1** | **UI 测试** | 页面能打开、布局/样式符合预期、关键元素存在且可交互 | 浏览器自动化（如 Playwright）或快照/可访问性工具 |
| **2** | **功能测试** | 完整用户流程可用：上传 → 解析 EXIF → 地图/元数据展示 → 剥离/导出 | 同上浏览器 E2E，或带真实 Provider 的集成测试 |
| **3** | **代码测试** | 单函数/单模块行为正确、重构不破坏逻辑 | 单元测试（Vitest/Jest）+ 少量 Hook/组件测试 |

下面分别说明各阶段写什么、用什么写。

---

## 三、阶段一：UI 测试

### 3.1 目的

- 各路由能正常渲染、无白屏/报错。
- 导航、Footer、多语言切换等关键 UI 存在且可点击。
- 上传区、Metadata 区块、地图占位等关键区块在未上传时展示符合预期（如「No metadata yet.」）。

### 3.2 推荐方式

- **Playwright**（与 Next 配合良好）：启动本地 dev 或 build 后，用 `page.goto("/")`、`page.getByRole(...)` 等做断言。
- 可选：**视觉回归**（截图对比）或 **可访问性**（axe）作为 UI 测试的一部分。

### 3.3 建议用例（示例）

- 首页 `/` 加载成功，且包含 Hero、上传区、FAQ 等区块。
- 导航可点击并跳转到 `/privacy`、`/terms`、`/about`、`/faq`。
- 语言切换后，页面文案或 `lang` 属性发生变化。
- 未上传时，Metadata 区域显示「No metadata yet.」及描述文案。

### 3.4 实施要点

- 在 `package.json` 增加 `"test:e2e"` 或 `"test:ui"`，指向 Playwright。
- 可先只测「能打开、关键元素存在」，再逐步加样式/可访问性断言。

---

## 四、阶段二：功能测试

### 4.1 目的

- 从用户视角验证：选图 → 解析 EXIF → 看到元数据/地图 → 选择预设剥离 → 导出（CSV/JSON/TXT/Hex）等流程畅通。
- 不关心内部函数实现，只关心「功能是否可用」。

### 4.2 推荐方式

- **Playwright E2E**：真实浏览器、真实文件上传（用测试用图片 fixture）、真实点击「Choose files」「Strip」「Export」等，断言页面状态或下载结果。
- 需要准备：少量带 EXIF 的测试图片（如含 GPS、设备信息），用于上传与断言。

### 4.3 建议用例（示例）

- 上传一张带 EXIF 的 JPEG → 页面显示元数据区块、地图有标点（或至少无报错）。
- 选择「隐私」预设并执行剥离 → 得到新文件；可再上传新文件断言 GPS/设备相关 tag 被去掉（或仅断言流程完成）。
- 导出 CSV/JSON → 检查下载的 blob 或文件名、内容格式是否符合预期。

### 4.4 测试图片存放位置

- **目录**：`e2e/fixtures/images/`
- 建议至少放一张带 EXIF 的 JPEG（如 `with-exif.jpg`），用于「上传 → 解析 → 元数据/地图展示」等用例；可选一张无 EXIF 的图（如 `no-exif.jpg`）。详见该目录下 `README.md`。
- 测试中通过 `path.join(process.cwd(), 'e2e/fixtures/images/xxx.jpg')` 或 `require('path').join(__dirname, 'fixtures/images/xxx.jpg')` 引用。

### 4.5 实施要点

- 与 UI 测试共用 Playwright 配置，用例放在 `e2e/` 下（如 `e2e/functional.spec.ts`）。
- 上传用 `setInputFiles` 注入上述 fixture 路径；导出可断言 `download` 事件或临时读文件内容。

## 五、阶段三：代码测试（单元 / 集成）

在 UI 与功能都有一层保障后，再针对「代码本身」写单测与少量集成测。

### 5.1 推荐框架

| 方案 | 适用 |
|------|------|
| **Vitest** | 纯函数、hook、工具函数；启动快、与 ESM 友好。 |
| **Jest + next/jest** | 若需大量测含 `next/router`、`next/image` 的组件时可选用。 |

### 5.2 纯函数 / 纯数据（优先）

| 文件 | 可测内容 | 说明 |
|------|----------|------|
| **stripExifPresets.ts** | `isPresetSupportedForFormat(mime)` | 无依赖；测 jpeg/jpg、大小写、非法 MIME。 |
| **stripExifPresets.ts** | `PRESET_TO_BLOCKS` 等 | 快照或简单断言。 |
| **parseExif.ts** | `getTagDisplayValue`、`getSignedLatLngFromTags`、`fileInfoFromFile`、`fileInfoFromTags`、`blocksFromTags` | 构造 `ExifTags`/`File` fixture 即可。 |
| **exportExif.ts** | `buildExifExportContent(blocks, format, filenameBase)` | 测 csv/json/txt 的 blob、文件名、内容格式。 |

### 5.3 需 mock 的异步 / 外部依赖

| 文件 | 函数 | Mock | 测试重点 |
|------|------|------|----------|
| **parseExif.ts** | `loadExifTags(file)` | ExifReader.load | 成功/失败返回形态。 |
| **exportExif.ts** | `buildExifExportHex(file)` | File / 解析结果 | 能调通、返回 blob。 |
| **stripExif.ts** | `stripExifFromFile(file, preset)` | piexifjs、File | 或先测拆出的纯逻辑。 |

### 5.4 自定义 Hook

- **useUploadFiles**：`renderHook` + mock `File`、`URL.createObjectURL`/`revokeObjectURL`；测 `appendFiles`、`clear`、`selectedId`/`selectedItem`、不支持格式被过滤。

### 5.5 组件（可选）

- 含 `useTranslations`：包一层 Provider 或 mock LocaleContext。
- 含 `useUploadContext`：用 UploadProvider + 假数据。
- Leaflet 地图：浅渲染或跳过，避免 DOM/地图库依赖。

### 5.6 实施要点

- 配置 Vitest：`testEnvironment: "jsdom"`、`include: ["src/**/*.test.ts", "src/**/*.test.tsx"]`。
- `package.json`：`"test": "vitest"`、`"test:run": "vitest run"`。
- 先写 stripExifPresets、parseExif、exportExif 的纯函数用例，再补 loadExifTags、useUploadFiles。

---

## 六、目录与脚本建议

- **UI / 功能**：`e2e/*.spec.ts` 或 `tests/e2e/*.spec.ts`（Playwright）。
- **代码**：`src/**/*.test.ts` 或 `src/__tests__/**`（Vitest/Jest）。
- **package.json 脚本示例**：
  - `"test:ui"` 或 `"test:e2e"`：Playwright（先 UI，再功能）。
  - `"test"`：Vitest watch；`"test:run"`：单次运行，用于 CI。

按「UI 测试 → 功能测试 → 代码测试」顺序推进，即可先保证页面与流程可用，再逐步夯实底层逻辑与重构安全。
